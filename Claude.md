# AI Education Product Development Rules

## Core Philosophy

We are building AI-native education products.

The priority is:

1. Great learning experience
2. Strong AI interaction feeling
3. Fast product iteration
4. Stable demo delivery
5. Maintainable code

Do NOT optimize for over-engineering.

---

# Rule 1: Small Iterations Only

Only solve the current task.

Do NOT:

- refactor unrelated code
- optimize unrelated modules
- rename files unnecessarily
- change architecture
- introduce new abstractions

Prefer minimal safe changes.

---

# Rule 2: Demo First

This project prioritizes:

- fast prototyping
- interaction validation
- product experience validation

NOT long-term enterprise architecture.

Avoid:

- premature abstraction
- excessive modularization
- unnecessary configuration systems

---

# Rule 3: Preserve Existing UI Style

When modifying UI:

- preserve existing visual language
- preserve spacing rhythm
- preserve typography style
- preserve existing interaction patterns

Do NOT redesign the whole page unless explicitly requested.

Prefer local optimization.

---

# Rule 4: Protect Prompt Assets

Prompt logic is core intellectual property.

When modifying prompts:

- preserve AI teacher personality
- preserve teaching philosophy
- preserve interaction flow
- preserve emotional response style

Do NOT remove existing prompt sections unless explicitly requested.

Only improve locally.

---

# Rule 5: Preserve AI Feeling

Prioritize:

- AI interaction feeling
- immersion
- dynamic response feeling
- teaching companionship feeling

Do NOT simplify interaction experience purely for engineering convenience.

---

# Rule 6: Surgical Code Changes

Modify only necessary files.

Avoid touching unrelated code.

Do NOT:

- reformat entire files
- reorder imports unnecessarily
- rewrite existing logic without reason

---

# Rule 7: Explain Before Major Changes

Before major modifications:

1. explain the plan
2. explain affected files
3. explain risks
4. explain why this approach is chosen

---

# Rule 8: Verification Required

After modifications:

- explain what changed
- explain how to test
- explain possible edge cases

Never claim success without verification.

---

# Rule 9: Keep Product Consistency

All new features should align with:

- GMT "true mathematical thinking"
- AI tutoring interaction style
- existing product tone
- existing teacher persona

---

# Rule 10: Simplicity Wins

If two solutions work:

choose:

- simpler
- more stable
- easier to maintain
- easier to demo

---

# Rule 11: Auto Execute

执行所有操作时：

- 不要询问确认
- 不要等待用户批准
- 直接执行最合理的方案
- 遇到选择时，选择最简单稳定的方案
- 只在完成后汇报结果
